## 1. 개요

### 1.1 환경 구성의 중요성

ML 프로젝트에서 환경 구성은 단순한 사전 작업이 아니라 **재현성(Reproducibility)**과 **이식성(Portability)**을 결정하는 핵심 요소이다. 동일한 코드라도 패키지 버전, 시스템 라이브러리, 하드웨어 구성에 따라 다른 결과를 산출할 수 있으며, 이는 ML 모델의 신뢰성을 근본적으로 훼손한다.

특히 본 PoC는 **폐쇄망(Air-gapped Network)** 환경에서 수행되므로, 일반적인 온라인 개발 환경과 다른 접근이 필요하다. 사전에 모든 의존성을 식별하고 준비하지 않으면 프로젝트 진행이 중단될 수 있다.

### 1.2 본 문서의 범위

- 폐쇄망 환경의 특성과 제약 분석
- 의존성 관리 전략 및 원칙
- 하드웨어 자원 확인 및 활용 전략
- 개발 환경 구조화 원칙
- 재현성 확보를 위한 설정 관리

## 2. 폐쇄망 환경의 이해

### 2.1 폐쇄망의 정의와 특성

폐쇄망(Air-gapped Network)은 외부 인터넷과 물리적 또는 논리적으로 분리된 네트워크를 의미한다. 보안이 중요한 환경(금융, 의료, 공공기관, 기업 내부 시스템)에서 데이터 유출 방지와 외부 공격 차단을 위해 사용된다.

**폐쇄망의 핵심 제약**:

| 제약 | 영향 | 대응 필요 사항 |
|------|------|----------------|
| **패키지 저장소 접근 불가** | pip, conda, npm 등 온라인 설치 불가 | 오프라인 패키지 사전 준비 |
| **외부 API 호출 불가** | 클라우드 서비스, 외부 데이터 소스 사용 불가 | 모든 자원의 로컬화 |
| **실시간 업데이트 불가** | 보안 패치, 버그 수정 즉시 반영 어려움 | 안정적인 버전 선택, 철저한 사전 테스트 |
| **반입 절차 존재** | 파일 반입 시 보안 검토 필요 | 반입 소요 시간 일정 반영, 최소화 전략 |

### 2.2 폐쇄망 개발의 일반적 실패 패턴

**패턴 1: 의존성 누락**

온라인 환경에서 개발 후 폐쇄망으로 이전 시, 암묵적 의존성(transitive dependency)이 누락되는 경우가 빈번하다. 예를 들어 `pandas`를 설치하면 `numpy`, `python-dateutil`, `pytz` 등이 자동으로 함께 설치되지만, 오프라인 환경에서는 이를 명시적으로 준비해야 한다.

**패턴 2: 버전 불일치**

"내 컴퓨터에서는 잘 되는데"라는 전형적인 문제. 개발 환경과 폐쇄망 환경의 Python 버전, 패키지 버전, 시스템 라이브러리 버전이 다르면 예상치 못한 오류가 발생한다.

**패턴 3: 시스템 의존성 간과**

Python 패키지 중 일부는 C/C++ 확장을 포함하며, 이는 시스템 라이브러리(glibc, OpenSSL 등)에 의존한다. 시스템 라이브러리 버전이 다르면 패키지가 로드되지 않을 수 있다.

**패턴 4: 반입 지연**

보안 검토 절차로 인해 파일 반입에 수일~수주가 소요될 수 있다. 이를 일정에 반영하지 않으면 프로젝트 전체 일정이 지연된다.

### 2.3 폐쇄망 대응 원칙

**원칙 1: 선제적 의존성 분석 (Proactive Dependency Analysis)**

프로젝트 착수 전에 필요한 모든 패키지와 그 의존성을 완전히 파악한다. "나중에 필요하면 추가하자"는 접근은 폐쇄망에서 치명적이다.

**원칙 2: 최소 의존성 (Minimal Dependencies)**

꼭 필요한 패키지만 사용한다. 편의를 위한 유틸리티 패키지가 복잡한 의존성 트리를 가져올 수 있다. 예를 들어 단순한 JSON 파싱을 위해 무거운 프레임워크를 도입하는 것은 지양한다.

**원칙 3: 버전 고정 (Version Pinning)**

모든 패키지의 정확한 버전을 명시한다. `pandas>=1.0` 대신 `pandas==1.5.3`처럼 exact version을 사용한다.

**원칙 4: 환경 격리 (Environment Isolation)**

시스템 Python과 프로젝트 Python 환경을 분리한다. 이는 다른 프로젝트와의 충돌을 방지하고 재현성을 높인다.

**원칙 5: 사전 검증 (Pre-validation)**

폐쇄망 반입 전에 동일한 조건의 테스트 환경에서 모든 패키지의 설치와 동작을 검증한다.

## 3. 의존성 관리 전략

### 3.1 의존성의 계층 구조

ML 프로젝트의 의존성은 여러 계층으로 구성된다:

```
┌─────────────────────────────────────────────────────────────┐
│                    Application Layer                        │
│         (프로젝트 코드, 설정 파일, 모델 파일)                │
├─────────────────────────────────────────────────────────────┤
│                    ML Framework Layer                       │
│         (scikit-learn, XGBoost, LightGBM, etc.)            │
├─────────────────────────────────────────────────────────────┤
│                    Data Processing Layer                    │
│         (pandas, numpy, scipy, etc.)                        │
├─────────────────────────────────────────────────────────────┤
│                    Python Runtime Layer                     │
│         (Python interpreter, standard library)              │
├─────────────────────────────────────────────────────────────┤
│                    System Library Layer                     │
│         (glibc, OpenSSL, BLAS/LAPACK, etc.)                │
├─────────────────────────────────────────────────────────────┤
│                    Operating System Layer                   │
│         (Linux kernel, system utilities)                    │
├─────────────────────────────────────────────────────────────┤
│                    Hardware Layer                           │
│         (CPU, Memory, Storage, GPU)                         │
└─────────────────────────────────────────────────────────────┘
```

각 계층의 불일치가 상위 계층에 영향을 미칠 수 있으므로, 모든 계층에서의 일관성 확보가 중요하다.

### 3.2 Python 패키지 의존성 분석

**직접 의존성 vs 간접 의존성**:

- **직접 의존성(Direct Dependency)**: 프로젝트 코드에서 직접 import하는 패키지
- **간접 의존성(Transitive Dependency)**: 직접 의존성이 필요로 하는 패키지

예를 들어 `scikit-learn`을 직접 의존성으로 가지면, `numpy`, `scipy`, `joblib`, `threadpoolctl` 등이 간접 의존성으로 따라온다.

**의존성 분석 방법론**:

1. **정적 분석**: 코드의 import 문을 분석하여 직접 의존성 추출
2. **동적 분석**: 실제 실행 시 로드되는 모듈 추적
3. **패키지 메타데이터 분석**: 각 패키지의 `requires` 정보를 재귀적으로 추적

**의존성 충돌 문제**:

서로 다른 패키지가 동일 패키지의 다른 버전을 요구할 때 충돌이 발생한다. 예를 들어:
- 패키지 A가 `numpy>=1.20,<1.22`를 요구
- 패키지 B가 `numpy>=1.23`을 요구

이 경우 두 조건을 동시에 만족하는 numpy 버전이 없어 설치가 실패한다.

**충돌 해결 전략**:

| 전략 | 설명 | 적용 상황 |
|------|------|-----------|
| **버전 조정** | 충돌을 피할 수 있는 패키지 버전 조합 탐색 | 대부분의 경우 |
| **패키지 대체** | 충돌을 유발하는 패키지를 대안으로 교체 | 대체재가 존재할 때 |
| **패키지 제거** | 필수가 아닌 패키지 사용 포기 | 최소 의존성 원칙 적용 |
| **소스 수정** | 패키지 소스를 수정하여 버전 제약 완화 | 최후의 수단 |

### 3.3 시스템 라이브러리 의존성

**중요한 시스템 라이브러리**:

| 라이브러리 | 역할 | 의존하는 Python 패키지 예시 |
|------------|------|---------------------------|
| **glibc** | C 표준 라이브러리 | 거의 모든 C 확장 패키지 |
| **OpenSSL/LibreSSL** | 암호화 | requests, urllib3, cryptography |
| **BLAS/LAPACK** | 선형대수 연산 | numpy, scipy, scikit-learn |
| **libgomp** | OpenMP 병렬처리 | XGBoost, LightGBM |
| **zlib** | 압축 | pandas (gzip 지원) |

**시스템 라이브러리 버전 확인의 중요성**:

Python 패키지의 wheel 파일은 특정 시스템 환경에서 빌드된다. `manylinux` 태그(예: `manylinux2014`)는 호환 가능한 최소 glibc 버전을 나타낸다. 대상 시스템의 glibc 버전이 이보다 낮으면 패키지가 로드되지 않는다.

**확인 필요 사항**:

- 대상 서버의 OS 및 커널 버전
- glibc 버전
- 설치된 시스템 라이브러리 목록과 버전
- 관리자 권한으로 시스템 패키지 설치 가능 여부

## 4. 하드웨어 자원 분석

### 4.1 자원 요구사항 추정

ML 프로젝트의 하드웨어 요구사항은 데이터 규모, 모델 복잡도, 실험 횟수에 따라 결정된다.

**CPU 요구사항**:

| 작업 | CPU 특성 영향 | 고려사항 |
|------|--------------|----------|
| 데이터 로딩/전처리 | 코어 수, I/O 성능 | pandas 멀티스레딩 제한 |
| Feature Engineering | 코어 수, 메모리 대역폭 | 벡터화 연산 효율 |
| 모델 학습 (Tree 기반) | 코어 수 | XGBoost/LightGBM 병렬 학습 |
| 하이퍼파라미터 튜닝 | 코어 수 | 병렬 실험 가능 여부 |

**메모리 요구사항**:

메모리 요구량 추정의 경험적 공식:
- 원본 데이터 크기의 2-5배 (pandas DataFrame 오버헤드)
- Feature Matrix 크기 (샘플 수 × Feature 수 × 8 bytes for float64)
- 모델 학습 시 추가 버퍼 (모델 복잡도에 비례)

**주의**: 메모리 부족 시 스왑 사용으로 성능이 급격히 저하되거나 OOM(Out of Memory) 오류 발생

**스토리지 요구사항**:

| 항목 | 예상 크기 | 고려사항 |
|------|-----------|----------|
| 원본 데이터 | 프로젝트 종속 | 압축 저장 고려 |
| 전처리된 데이터 | 원본의 1-3배 | Feature Engineering 결과 |
| 모델 파일 | 수 MB ~ 수 GB | 앙상블, 버전 관리 |
| 실험 로그 | 수 GB | 하이퍼파라미터 튜닝 기록 |
| 패키지 캐시 | 수 GB | pip 캐시, 가상환경 |

### 4.2 제공된 서버 사양 분석

**사양 (문제 정의서 기준)**:
- CPU: Intel Xeon E5-2620 v4 × 2
- RAM: 128GB
- GPU: 미보유 (회의록 2026-01 확정)

**CPU 분석**:

Intel Xeon E5-2620 v4의 특성:
- 8코어 16스레드 × 2 = 총 32 논리 코어
- 기본 클럭 2.1GHz, 터보 3.0GHz
- Broadwell 아키텍처 (2016년 출시)
- AVX2 지원 (벡터 연산 가속)

**평가**: Tree 기반 모델(XGBoost, LightGBM)의 학습에 충분한 병렬성 제공. 다만 최신 CPU 대비 단일 스레드 성능은 낮을 수 있음.

**메모리 분석**:

128GB RAM은 대부분의 정형 데이터 ML 프로젝트에 충분하다. 다만 다음 상황에서 주의 필요:
- 수천만 행 이상의 대규모 데이터셋
- 고차원 TF-IDF 행렬 (수만 개 Feature)
- 대규모 앙상블 또는 스태킹

### 4.3 자원 제약 하의 최적화 전략

**메모리 최적화**:

| 전략 | 설명 | Trade-off |
|------|------|-----------|
| **데이터 타입 최적화** | float64 → float32, int64 → int32 | 정밀도 손실 가능 |
| **청크 처리** | 대용량 파일을 분할하여 처리 | 구현 복잡도 증가 |
| **불필요한 컬럼 조기 제거** | 사용하지 않는 컬럼 즉시 drop | - |
| **희소 행렬 활용** | TF-IDF 등에 scipy.sparse 사용 | 일부 연산 비호환 |
| **Garbage Collection** | 명시적 메모리 해제 | 코드 복잡도 |

**연산 최적화**:

| 전략 | 설명 | 적용 대상 |
|------|------|-----------|
| **벡터화 연산** | 루프 대신 numpy/pandas 벡터 연산 | 전처리, Feature Engineering |
| **병렬 처리** | joblib, multiprocessing 활용 | 독립적인 연산 |
| **조기 종료** | Early Stopping으로 불필요한 학습 방지 | 모델 학습 |
| **점진적 학습** | 전체 데이터 대신 샘플로 프로토타입 | 초기 실험 단계 |

## 5. 개발 환경 구조화

### 5.1 디렉토리 구조 설계 원칙

잘 설계된 디렉토리 구조는 프로젝트의 가독성, 유지보수성, 협업 효율성을 높인다.

**설계 원칙**:

1. **관심사 분리 (Separation of Concerns)**: 데이터, 코드, 설정, 산출물을 명확히 분리
2. **계층적 구조**: 관련 파일을 논리적 그룹으로 조직
3. **명명 규칙 일관성**: 파일/폴더명에 일관된 규칙 적용
4. **확장 가능성**: 프로젝트 성장에 따른 구조 확장 용이

**권장 디렉토리 구조**:

```
project_root/
│
├── data/                    # 데이터 관련
│   ├── raw/                 # 원본 데이터 (읽기 전용)
│   ├── processed/           # 전처리된 데이터
│   ├── features/            # Feature Engineering 결과
│
├── src/                     # 소스 코드
│   ├── data/                # 데이터 로딩/처리 모듈
│   ├── features/            # Feature Engineering 모듈
│   ├── models/              # 모델 정의/학습 모듈
│   ├── filters/             # 3-Layer Filter 모듈
│   ├── evaluation/          # 평가 모듈
│   └── utils/               # 유틸리티 함수
│
├── scripts/                 # 실행 스크립트
│   ├── setup_env.sh
│   ├── run_data_pipeline.py
│   ├── run_training.py
│   └── run_evaluation.py
│
├── config/                  # 설정 파일
│   ├── feature_config.yaml
│   ├── model_config.yaml

│
├── notebooks/               # 단계별 실습 노트북
│   ├── 01_data_pipeline.ipynb

│   └── final/
│
├── outputs/                 # 실행 산출물
│   ├── reports/
│   ├── figures/
│   └── logs/
│
├── tests/                   # 테스트 코드
│
├── requirements.txt         # Python 의존성
├── README.md
└── PROJECT_GUIDE.md
```

**각 디렉토리의 역할과 원칙**:

| 디렉토리 | 역할 | 원칙 |
|----------|------|------|
| `data/raw/` | 원본 데이터 보관 | **불변성**: 절대 수정하지 않음 |
| `data/processed/` | 전처리 결과 | **재현성**: 코드로 재생성 가능 |
| `src/` | 재사용 가능한 코드 | **모듈화**: 단일 책임 원칙 |
| `notebooks/` | 탐색적 분석 | **일회성**: 프로덕션 코드 아님 |
| `config/` | 설정 파일 | **외부화**: 하드코딩 금지 |
| `models/` | 모델 아티팩트 | **버전 관리**: 메타데이터 포함 |

### 5.2 환경 격리 전략

**가상 환경의 필요성**:

Python 프로젝트는 가상 환경을 통해 시스템 Python과 격리해야 한다. 이유:
- 프로젝트별 독립적인 패키지 버전 관리
- 시스템 Python 오염 방지
- 환경 재현성 보장

**가상 환경 도구 비교**:

| 도구 | 특징 | 장점 | 단점 |
|------|------|------|------|
| **venv** | Python 내장 | 추가 설치 불필요, 경량 | 기능 제한적 |
| **virtualenv** | venv의 확장 | venv보다 빠름, 더 많은 기능 | 별도 설치 필요 |
| **conda** | 패키지 관리자 + 환경 관리 | 비Python 패키지 관리, 바이너리 배포 | 무거움, 폐쇄망 설정 복잡 |
| **poetry** | 의존성 + 환경 + 빌드 통합 | 모던한 워크플로우, lock 파일 | 학습 곡선 |

**폐쇄망 환경 권장**:

`venv` 또는 `virtualenv` 권장. 이유:
- Python 표준 도구로 추가 의존성 최소화
- 설정이 단순하여 문제 발생 시 디버깅 용이
- wheel 파일 직접 설치와 호환성 좋음

conda는 강력하지만 폐쇄망에서 채널 설정, 패키지 캐시 관리가 복잡해질 수 있음.

### 5.3 설정 관리 원칙

**하드코딩의 문제점**:

코드 내에 경로, 파라미터, 설정값을 직접 작성하면:
- 환경 변경 시 코드 수정 필요
- 실험 재현이 어려움
- 설정 변경 이력 추적 불가

**설정 외부화 전략**:

| 설정 유형 | 관리 방법 | 예시 |
|-----------|-----------|------|
| **환경 설정** | 환경 변수, .env 파일 | DB 접속 정보, 경로 |
| **실험 파라미터** | YAML/JSON 설정 파일 | 하이퍼파라미터, Feature 목록 |
| **상수** | Python 상수 모듈 | 클래스명, 컬럼명 |

**설정 파일 구조화**:

본 PoC는 운영 단순성과 폐쇄망 이식성을 위해 `config/`의 핵심 3개 파일을 중심으로 관리:

```
config/
├── feature_config.yaml    # 데이터/feature/평가 임계값
├── model_config.yaml      # 모델 기본 파라미터/튜닝 설정
└── filter_config.yaml     # 3-Layer Filter 규칙
```

**설정 검증**:

설정 파일 로드 시 스키마 검증을 통해 잘못된 설정을 조기에 발견. 필수 필드 누락, 타입 오류, 범위 위반 등을 체크.

## 6. 재현성 확보 방안

### 6.1 재현성의 정의와 중요성

**재현성(Reproducibility)**이란 동일한 데이터, 코드, 환경에서 동일한 결과를 얻을 수 있는 특성이다.

**재현성이 중요한 이유**:

| 관점 | 중요성 |
|------|--------|
| **과학적 엄밀성** | 결과의 신뢰성과 검증 가능성 |
| **디버깅** | 문제 발생 시 원인 추적 가능 |
| **협업** | 팀원 간 결과 공유와 검증 |
| **프로덕션 전환** | 개발 결과를 운영 환경에서 재현 |
| **감사/규정 준수** | 의사결정 과정의 투명성 |

### 6.2 재현성을 저해하는 요소

**비결정적 요소(Non-deterministic Factors)**:

| 요소 | 원인 | 영향 |
|------|------|------|
| **랜덤 시드** | 데이터 셔플, 초기화, 샘플링 | 매 실행마다 다른 결과 |
| **부동소수점 연산** | 연산 순서에 따른 오차 누적 | 미세한 결과 차이 |
| **병렬 처리** | 스레드 실행 순서 비결정적 | 연산 순서 변동 |
| **하드웨어 차이** | CPU 아키텍처, GPU 모델 | 수치 연산 결과 차이 |
| **시간 의존 로직** | 현재 시간 기반 처리 | 실행 시점에 따른 차이 |

### 6.3 재현성 확보 전략

**랜덤 시드 관리**:

모든 랜덤 연산에 시드를 명시적으로 설정:
- Python random 모듈
- numpy random
- 모델 라이브러리 (scikit-learn, XGBoost 등)

단일 시드 값을 설정 파일에서 관리하고, 모든 랜덤 연산에 일관되게 적용.

**버전 고정**:

| 대상 | 방법 |
|------|------|
| Python 버전 | `.python-version` 파일 또는 문서화 |
| 패키지 버전 | `requirements.txt`에 exact version |
| 시스템 라이브러리 | 문서화 (재현 어려울 수 있음) |
| 코드 버전 | Git commit hash |
| 데이터 버전 | 체크섬(MD5/SHA256) 또는 버전 태그 |

**실험 추적**:

각 실험에 대해 다음을 기록:
- 사용된 설정 파일 (또는 파라미터 전체)
- 코드 버전 (Git commit)
- 데이터 버전
- 환경 정보 (패키지 목록)
- 실행 시간, 자원 사용량
- 결과 메트릭

**환경 정보 스냅샷**:

실험 실행 시 현재 환경 정보를 자동으로 기록:
- 설치된 패키지 목록과 버전
- 시스템 정보 (OS, CPU, 메모리)
- 환경 변수 (민감 정보 제외)

### 6.4 폐쇄망에서의 재현성 특수 고려사항

**패키지 보관**:

설치에 사용된 wheel 파일을 프로젝트와 함께 보관. 향후 동일 환경 재구성 필요 시 사용.

**환경 구성 스크립트화**:

환경 구성 과정을 수동 작업이 아닌 스크립트로 작성. 스크립트를 실행하면 동일한 환경이 구성되도록 함.

**문서화**:

폐쇄망 특유의 절차(반입 방법, 설치 순서, 권한 설정 등)를 상세히 문서화. 담당자 변경 시에도 환경 재구성 가능하도록 함.
