## 1. 개요

### 1.1 EDA의 정의와 목적

탐색적 데이터 분석(Exploratory Data Analysis, EDA)은 통계학자 John Tukey가 1977년 제안한 개념으로, 가설 검정이나 모델 구축에 앞서 데이터 자체를 다양한 각도에서 탐색하여 그 특성을 이해하는 과정이다.

**EDA의 핵심 철학**:

Tukey는 "데이터가 말하게 하라(Let the data speak)"고 강조했다. 이는 선입견이나 가정 없이 데이터 자체에서 패턴과 이상을 발견해야 함을 의미한다. ML 프로젝트에서 EDA는 단순한 사전 작업이 아니라, 전체 모델링 전략을 결정짓는 핵심 단계이다.

**EDA를 통해 답해야 할 질문들**:

| 영역 | 질문 | 모델링에 미치는 영향 |
|------|------|---------------------|
| **데이터 품질** | 결측치, 이상치, 중복은 얼마나 있는가? | 전처리 전략 결정 |
| **클래스 분포** | 클래스별 샘플 수와 비율은? | 불균형 대응 전략 |
| **Feature 특성** | 각 Feature의 분포, 범위, 유형은? | Feature Engineering 방향 |
| **변수 간 관계** | Feature와 Target의 관계는? Feature 간 상관은? | Feature Selection, 다중공선성 |
| **데이터 누수** | Target 정보가 Feature에 누출되어 있지 않은가? | 모델 신뢰성 |

### 1.2 EDA의 단계적 접근

EDA는 일반적으로 다음 순서로 진행된다:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           EDA 수행 흐름                                     │
└─────────────────────────────────────────────────────────────────────────────┘

[1단계: 데이터 개요 파악]
    │
    ├── 데이터 규모 (행/열 수)
    ├── 데이터 타입 분포
    └── 기본 통계량
    │
    ▼
[2단계: 단변량 분석]
    │
    ├── Target 변수 (클래스 분포)
    ├── 수치형 Feature (분포, 이상치)
    ├── 범주형 Feature (빈도, 카디널리티)
    └── 텍스트 Feature (길이, 패턴)
    │
    ▼
[3단계: 이변량/다변량 분석]
    │
    ├── Feature-Target 관계
    ├── Feature 간 상관관계
    └── 그룹별 패턴 차이
    │
    ▼
[4단계: 데이터 품질 심층 분석]
    │
    ├── 결측치 패턴
    ├── 이상치 분포
    └── 데이터 누수 점검
    │
    ▼
[5단계: 인사이트 종합 및 전략 수립]
    │
    └── 모델링 전략, Feature Engineering 방향 도출
```

### 1.3 본 PoC에서의 EDA 특수 고려사항

Server-i 오탐 개선 프로젝트는 다음과 같은 특수성을 가진다:

| 특수성 | EDA 시 고려사항 |
|--------|----------------|
| **극심한 클래스 불균형** | 전체 분포뿐 아니라 소수 클래스(정탐)의 특성 집중 분석 |
| **멀티모달 데이터** | Text와 Tabular 각각의 특성 및 상호작용 분석 |
| **Multi-class 구조** | 오탐 하위 클래스 간 구분 가능성 분석 |
| **도메인 특수성** | 파일 경로, PII 패턴 등 도메인 지식 활용 |

## 2. 데이터 개요 분석

### 2.1 데이터 규모 파악

**확인해야 할 항목**:

| 항목 | 확인 내용 | 의미 |
|------|-----------|------|
| **전체 레코드 수** | 학습에 사용 가능한 샘플 수 | 모델 복잡도 상한, 통계적 유의성 |
| **Feature 수** | 원본 Feature 개수 | Feature Engineering 범위 |
| **메모리 사용량** | 데이터프레임 크기 | 처리 전략 (전체 로드 vs 청크) |
| **파일 크기** | 원본 데이터 용량 | 저장/전송 고려 |

**규모에 따른 전략 분기**:

| 규모 | 기준 | 전략적 고려 |
|------|------|-------------|
| **소규모** | < 10,000 | 과적합 위험, 단순 모델 선호, Cross-Validation 필수 |
| **중규모** | 10,000 - 100,000 | 균형 잡힌 접근, 다양한 모델 실험 가능 |
| **대규모** | > 100,000 | 샘플링 기반 EDA, 메모리 효율화 필요 |

### 2.2 데이터 타입 분포

데이터의 타입 구성을 파악하여 처리 전략을 수립한다.

**타입별 분류**:

| 타입 | 예시 필드 | 처리 방향 |
|------|-----------|-----------|
| **수치형 연속** | 파일 크기, 텍스트 길이 | 스케일링, 분포 변환 |
| **수치형 이산** | 검출 횟수, 경로 깊이 | 연속형 또는 범주형으로 처리 |
| **범주형 명목** | PII 유형, 확장자 | 인코딩 |
| **범주형 순서** | 심각도 등급 (있는 경우) | 순서 보존 인코딩 |
| **텍스트** | 검출 텍스트, 파일 경로 | 텍스트 Feature Engineering |
| **날짜/시간** | 검출 일시 | 시간 Feature 추출 |

### 2.3 기본 통계량 검토

**수치형 변수의 기본 통계**:

| 통계량 | 의미 | 주의 깊게 볼 지점 |
|--------|------|-------------------|
| **count** | 유효 값 수 | 전체 행 수와의 차이 = 결측 |
| **mean, median** | 중심 경향 | 차이가 크면 왜도 존재 |
| **std** | 산포 | 평균 대비 매우 크면 이상치 의심 |
| **min, max** | 범위 | 비현실적 값 존재 여부 |
| **25%, 50%, 75%** | 분위수 | IQR 기반 이상치 탐지 |

## 3. 클래스 분포 분석

### 3.1 클래스 불균형의 정량화

**불균형 비율(Imbalance Ratio, IR) 계산**:

```
IR = (다수 클래스 샘플 수) / (소수 클래스 샘플 수)
```

본 PoC에서 정탐:오탐 ≈ 1:2라면 IR = 2이다.

**불균형 심각도 분류**:

| IR 범위 | 심각도 | 대응 필요성 |
|---------|--------|------------|
| 1-3 | 경미 | 기본적인 Stratified Sampling |
| 3-10 | 중간 | Class Weight 또는 오버샘플링 |
| 10-100 | 심각 | 복합적 불균형 대응 전략 |
| > 100 | 극심 | 이상 탐지 접근 고려 |

### 3.2 Multi-class 분포 분석

본 PoC는 8개 클래스를 가진 Multi-class 문제이다. Binary(정탐/오탐)뿐 아니라 세부 클래스의 분포도 분석해야 한다.

**분석 관점**:

| 관점 | 분석 내용 | 활용 |
|------|-----------|------|
| **절대 빈도** | 각 클래스의 샘플 수 | 희소 클래스 식별 |
| **상대 빈도** | 각 클래스의 비율 | 불균형 정도 파악 |
| **계층적 분포** | 정탐 vs 오탐, 오탐 내 분포 | 계층적 분류 가능성 |
| **Effective Number of Classes** | 실질적으로 유의미한 클래스 수 | 클래스 병합 필요성 |

**희소 클래스 처리 의사결정**:

특정 클래스의 샘플이 극히 적은 경우(예: 전체의 1% 미만, 절대 수 50개 미만):

| 옵션 | 설명 | 적용 상황 |
|------|------|-----------|
| **유지** | 그대로 학습 | 비즈니스적으로 중요한 클래스 |
| **병합** | 유사 클래스와 통합 | 의미적으로 유사한 클래스 존재 |
| **제외** | 학습에서 제외 | 노이즈성 데이터, 중요도 낮음 |
| **오버샘플링** | 합성 데이터 생성 | 클래스 구분이 명확한 경우 |

### 3.3 클래스별 특성 비교

각 클래스가 어떤 Feature에서 구분되는지 파악한다.

**비교 분석 항목**:

- 수치형 Feature: 클래스별 평균, 중앙값, 분포 비교
- 범주형 Feature: 클래스별 범주 분포 비교
- 텍스트 Feature: 클래스별 주요 키워드, 길이 분포

**클래스 분리도(Class Separability) 평가**:

특정 Feature가 클래스를 얼마나 잘 구분하는지 정성적으로 평가:
- 분포가 겹치지 않으면: 높은 분리도 → 유용한 Feature
- 분포가 완전히 겹치면: 낮은 분리도 → 해당 Feature만으로는 구분 어려움

## 4. 텍스트 데이터 분석

### 4.1 텍스트 분석의 중요성

본 PoC에서 텍스트 데이터(검출 텍스트 + 컨텍스트)는 정탐/오탐 구분의 핵심 정보를 담고 있을 가능성이 높다. 예를 들어:
- "테스트용 전화번호: 010-1234-5678" → 오탐(테스트) 시사
- "고객님의 연락처는 010-****-****입니다" → 정탐 가능성

### 4.2 텍스트 길이 분석

**분석 항목**:

| 항목 | 측정 단위 | 분석 목적 |
|------|-----------|-----------|
| **문자 수** | 글자 | 전체 길이 파악 |
| **단어 수** | 공백 기준 분리 | 텍스트 복잡도 |
| **토큰 수** | 형태소 분석 기반 | 실제 정보량 |

**클래스별 길이 분포 비교**:

길이가 클래스와 상관관계를 가지는지 확인:
- 예: 짧은 텍스트가 특정 오탐 유형에 집중되는지
- 예: 정탐이 특정 길이 범위에 분포하는지

**길이 이상치 분석**:

- 극단적으로 짧은 텍스트: 컨텍스트 부족으로 분류 어려움 예상
- 극단적으로 긴 텍스트: 노이즈 포함 가능성, 처리 비용 증가

### 4.3 어휘 분석

**빈도 기반 분석**:

| 분석 | 설명 | 활용 |
|------|------|------|
| **전체 어휘 빈도** | 가장 빈번한 단어/n-gram | 주요 키워드 파악 |
| **클래스별 어휘 빈도** | 클래스별 빈번 단어 | 클래스 특성 키워드 |
| **TF-IDF 상위 단어** | 문서 특이성 높은 단어 | 구분력 높은 키워드 |

**특수 패턴 분석**:

| 패턴 | 예시 | 의미 |
|------|------|------|
| **테스트/샘플 관련** | "test", "테스트", "sample", "예시" | FP-더미데이터 클래스 지표 |
| **코드/식별자 관련** | "debug", "TODO", "FIXME", 변수명 패턴 | FP-숫자나열/코드 클래스 지표 |
| **문서/맥락 관련** | "예)", "참고:", "출처:" | FP-패턴맥락 클래스 지표 |
| **시스템/시간 관련** | 로그 형식, 타임스탬프 패턴 | FP-타임스탬프 클래스 지표 |

### 4.4 한국어 텍스트 특수 고려사항

**한국어 텍스트의 특성**:

| 특성 | 영향 | 대응 |
|------|------|------|
| **교착어** | 어미 변화로 동일 어근 다른 형태 | 형태소 분석 필요 |
| **띄어쓰기 불규칙** | 토큰화 어려움 | 형태소 분석기 활용 |
| **한영 혼용** | 어휘 분리 필요 | 언어 감지 및 분리 처리 |
| **특수문자 혼합** | 이모지, 특수기호 | 정규화 전략 필요 |

**형태소 분석 고려사항**:

형태소 분석기(예: Mecab, Komoran)의 폐쇄망 설치 가능 여부 확인 필요. 설치가 어려운 경우:
- 공백 기반 단순 토큰화
- 문자 n-gram 사용
- 정규표현식 기반 패턴 매칭

### 4.5 원본 vs 마스킹 텍스트 비교

두 가지 텍스트 형태가 존재하므로 각각의 특성을 분석한다.

**비교 분석 항목**:

| 항목 | 원본 텍스트 | 마스킹 텍스트 |
|------|-------------|---------------|
| **정보량** | PII 값 자체의 패턴 확인 가능 | PII 값 비가시화 |
| **길이** | 실제 PII 길이 반영 | Asterisk로 대체된 길이 |
| **주변 컨텍스트** | 동일 | 동일 |
| **클래스 구분력** | 분석 필요 | 분석 필요 |

**의사결정 지원 분석**:

원본과 마스킹 각각에서 클래스별 구분 가능성을 분석하여, 어떤 데이터가 모델링에 더 적합한지 판단한다.

## 5. Tabular 데이터 분석

### 5.1 범주형 변수 분석

**분석 항목**:

| 항목 | 설명 | 고려사항 |
|------|------|----------|
| **카디널리티** | 고유 값의 수 | 고카디널리티 → 인코딩 전략 |
| **빈도 분포** | 각 범주의 출현 빈도 | 희소 범주 처리 |
| **클래스별 분포** | 범주와 Target의 관계 | 예측력 평가 |

**주요 범주형 변수 예상 분석**:

| 변수 | 분석 포인트 |
|------|-------------|
| **PII 유형** | 휴대폰/이메일/주민번호별 오탐 비율 차이 |
| **파일 확장자** | 확장자별 오탐 패턴 (예: .java → 개발 오탐 높음?) |
| **서버 그룹** | 개발/스테이징/운영 서버별 오탐 특성 |
| **검출 패턴** | 정규표현식 패턴별 정확도 차이 |

**고카디널리티 변수 처리 고려**:

파일 경로, 파일명 등은 고유 값이 매우 많을 수 있다(고카디널리티). 이 경우:
- 원본 그대로 사용 불가 (One-hot 인코딩 시 차원 폭발)
- 파생 변수 추출 필요 (경로 깊이, 특정 디렉토리 포함 여부 등)

### 5.2 수치형 변수 분석

**분포 분석**:

| 분포 특성 | 탐지 방법 | 처리 방향 |
|-----------|-----------|-----------|
| **정규 분포** | 히스토그램, Shapiro-Wilk 검정 | 스케일링 적용 |
| **왜도(Skewness)** | 왜도 계수, 히스토그램 | 로그/Box-Cox 변환 |
| **다봉 분포** | 히스토그램 | 그룹 분리 고려 |
| **이상치** | 박스플롯, Z-score | 이상치 처리 전략 |

**클래스별 분포 비교**:

수치형 변수의 분포가 클래스별로 다른지 확인:
- 분포 겹침 정도 → 구분력
- 평균/중앙값 차이 → 단순 임계값 기반 규칙 가능성

### 5.3 파일 경로 분석

파일 경로는 단순 문자열이지만 풍부한 구조적 정보를 담고 있다.

**추출 가능한 정보**:

| 정보 | 설명 | 예상 활용 |
|------|------|-----------|
| **경로 깊이** | 디렉토리 중첩 수준 | 시스템 파일 vs 사용자 파일 |
| **루트 디렉토리** | 최상위 디렉토리 | 파일 유형/용도 그룹화 |
| **특정 키워드 포함** | test, dev, backup, temp 등 | 오탐 유형 지표 |
| **파일명 패턴** | 숫자, 날짜, 버전 패턴 | 자동 생성 파일 식별 |

**경로 패턴과 클래스 관계 분석**:

예를 들어:
- `/home/*/test/*` 경로 → FP-더미데이터 비율이 높은지
- `/var/log/*` 경로 → FP-타임스탬프 또는 FP-패턴맥락 비율이 높은지
- `/home/*/Documents/*` 경로 → TP 비율이 높은지

### 5.4 시간 변수 분석

데이터에 시간 정보(검출 일시, 파일 수정 일시 등)가 포함된 경우:

**시간 기반 분석**:

| 분석 | 설명 | 활용 |
|------|------|------|
| **시간 분포** | 검출이 특정 시간대에 집중되는지 | 업무 시간 vs 비업무 시간 패턴 |
| **요일 분포** | 요일별 검출 패턴 | 주중 vs 주말 패턴 |
| **추세** | 시간에 따른 클래스 비율 변화 | Concept Drift 감지 |
| **계절성** | 주기적 패턴 | Feature로 활용 가능 |

**시간에 따른 분포 변화 (Concept Drift)**:

시간이 지남에 따라 데이터 분포나 클래스 비율이 변하는지 확인:
- 새로운 시스템 도입으로 새로운 오탐 패턴 등장
- 정책 변경으로 검출 기준 변화
- 계절적 업무 패턴 변화

이는 시간 기반 데이터 분할 여부와 모델 재학습 주기 결정에 영향을 준다.

## 6. 변수 간 관계 분석

### 6.1 Feature-Target 관계 분석

**분석 목적**:

각 Feature가 Target(클래스)과 얼마나 관련이 있는지 파악하여:
- 유용한 Feature 식별
- Feature Engineering 방향 수립
- 불필요한 Feature 제거 후보 식별

**분석 방법**:

| Feature 타입 | Target 타입 | 분석 방법 |
|--------------|-------------|-----------|
| 수치형 | 범주형 | 클래스별 분포 비교, ANOVA, Kruskal-Wallis |
| 범주형 | 범주형 | 교차표, Chi-square 검정, Cramér's V |
| 텍스트 | 범주형 | 클래스별 키워드 빈도, TF-IDF 분석 |

**주의사항**:

통계적 유의성과 실용적 유의성은 다르다. 대규모 데이터에서는 사소한 차이도 통계적으로 유의할 수 있다. 효과 크기(Effect Size)를 함께 고려해야 한다.

### 6.2 Feature 간 상관관계 분석

**상관관계 분석의 목적**:

| 목적 | 설명 |
|------|------|
| **다중공선성 탐지** | 높은 상관의 Feature 쌍 식별 |
| **중복 Feature 식별** | 거의 동일한 정보를 담는 Feature |
| **파생 변수 아이디어** | Feature 조합 가능성 |

**상관계수 해석**:

| 상관계수 범위 | 해석 |
|---------------|------|
| 0.9 - 1.0 | 매우 강한 상관 (다중공선성 위험) |
| 0.7 - 0.9 | 강한 상관 |
| 0.4 - 0.7 | 중간 상관 |
| 0.2 - 0.4 | 약한 상관 |
| 0.0 - 0.2 | 거의 무상관 |

**Tree 기반 모델에서의 다중공선성**:

일반적으로 다중공선성은 선형 모델에서 문제가 된다. Tree 기반 모델(XGBoost, LightGBM)은 다중공선성에 상대적으로 강건하다. 그러나:
- Feature Importance 해석이 어려워질 수 있음 (상관된 Feature 간 중요도 분산)
- 불필요한 Feature로 인한 과적합 위험

### 6.3 그룹별 패턴 분석

특정 그룹(Segment)에서 다른 패턴이 나타나는지 분석한다.

**분석 예시**:

| 그룹 기준 | 분석 내용 |
|-----------|-----------|
| **PII 유형별** | 휴대폰/이메일/주민번호별로 오탐 패턴이 다른가? |
| **서버 유형별** | 개발/운영 서버별로 클래스 분포가 다른가? |
| **파일 유형별** | 소스코드/문서/로그별로 Feature 분포가 다른가? |

**시사점**:

그룹별로 패턴이 크게 다르다면:
- 그룹을 별도 Feature로 활용
- 그룹별 별도 모델 고려 (본 PoC 범위 외)
- Slice-based 평가 필수

## 7. 데이터 누수(Data Leakage) 점검

### 7.1 Data Leakage의 정의와 위험성

**Data Leakage란**:

학습 데이터에 실제 예측 시점에는 알 수 없는 정보가 포함되어, 모델이 비현실적으로 높은 성능을 보이는 현상이다.

**위험성**:

- 오프라인 평가에서는 우수한 성능
- 실제 운영 환경에서 성능 급락
- 문제 발견이 늦어지면 프로젝트 실패

**Leakage의 유형**:

| 유형 | 설명 | 예시 |
|------|------|------|
| **Target Leakage** | Target에서 파생된 정보가 Feature에 포함 | 오탐으로 판정 후 추가된 태그 |
| **Train-Test Contamination** | Test 데이터 정보가 학습에 영향 | 전체 데이터로 스케일링 후 분할 |
| **Temporal Leakage** | 미래 정보로 과거 예측 | 검출 이후에 생성된 Feature |

### 7.2 Leakage 탐지 방법

**징후 기반 탐지**:

| 징후 | 의심 상황 |
|------|-----------|
| **비현실적으로 높은 성능** | 단순 모델로 F1 > 0.95 |
| **특정 Feature의 극단적 중요도** | 단일 Feature가 중요도 90% 이상 |
| **Feature 이름의 의심** | "label", "result", "판정" 등 포함 |
| **완벽한 분리** | 특정 Feature 값이 특정 클래스에만 존재 |

**체계적 점검 방법**:

1. **Feature 출처 검토**: 각 Feature가 언제, 어떻게 생성되는지 확인
2. **시간 순서 검증**: Feature 생성 시점이 예측 시점 이전인지 확인
3. **비즈니스 로직 검토**: 실제 운영 환경에서 해당 Feature를 사용할 수 있는지 확인

### 7.3 본 PoC에서의 Leakage 위험 지점

**주의해야 할 Feature들**:

| Feature 후보 | Leakage 위험 | 확인 필요 사항 |
|--------------|-------------|----------------|
| **처리 상태** | 높음 | 오탐 판정 후 업데이트되는 필드인지 |
| **담당자 지정 정보** | 높음 | 정탐 판정 후에만 지정되는지 |
| **파일 삭제 여부** | 높음 | 정탐 처리 후 삭제되는 패턴인지 |
| **검출 일시** | 낮음 | 예측 시점에 알 수 있는 정보 |
| **파일 메타데이터** | 낮음 | 검출 시점에 존재하는 정보 |

**검증 프로세스**:

1. 모든 Feature 목록 작성
2. 각 Feature의 생성 시점 및 방법 문서화
3. 예측 시점 기준으로 사용 가능 여부 판정
4. 의심 Feature 제거 후 모델 재학습하여 성능 변화 확인

### 7.4 Leakage 방지 원칙

**원칙 1: 시간 순서 엄수**

Feature는 반드시 예측 대상 시점 이전에 존재하는 정보만 사용해야 한다.

**원칙 2: 전처리 순서 준수**

데이터 분할 → 전처리 파라미터 계산(Train만) → 전처리 적용(Train, Val, Test)

잘못된 순서: 전처리(전체) → 데이터 분할

**원칙 3: Target 파생 정보 배제**

Target 레이블을 알아야만 계산할 수 있는 Feature는 사용 불가.

| Leakage 의심 Feature | 해당 Feature 제외 |
| 그룹별 패턴 차이 | Slice-based 평가 설계 |

### 8.3 반복적 EDA

**반복 시점**:

| **새 데이터 추가 시** | 분포 변화 모니터링 |
