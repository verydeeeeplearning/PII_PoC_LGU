## 1. 개요

### 1.1 데이터 파이프라인의 정의

데이터 파이프라인은 원본 데이터를 수집하여 모델이 소비할 수 있는 형태로 변환하고, 예측 결과를 비즈니스 프로세스에 연결하는 일련의 데이터 흐름을 의미한다. ML 프로젝트에서 데이터 파이프라인은 단순한 ETL(Extract-Transform-Load)을 넘어, 모델 학습과 추론의 일관성을 보장하는 핵심 인프라이다.

### 1.2 파이프라인 설계의 중요성

**"Garbage In, Garbage Out"** 원칙이 ML에서 특히 중요한 이유:

- 아무리 정교한 모델도 잘못된 데이터에서는 의미 있는 패턴을 학습할 수 없음
- 데이터 품질 문제는 모델 성능 저하의 가장 흔한 원인
- 학습 데이터와 추론 데이터의 불일치(Train-Serving Skew)는 프로덕션 실패의 주요 원인

**파이프라인 설계 시 달성해야 할 목표**:

| 목표 | 설명 |
|------|------|
| **정확성 (Correctness)** | 데이터 변환이 의도한 대로 정확히 수행됨 |
| **일관성 (Consistency)** | 학습과 추론에서 동일한 변환 로직 적용 |
| **추적 가능성 (Traceability)** | 모든 데이터의 출처와 변환 이력 추적 가능 |
| **재현성 (Reproducibility)** | 동일 입력에서 동일 출력 보장 |
| **확장성 (Scalability)** | 데이터 규모 증가에 대응 가능 |

### 1.3 본 PoC의 데이터 흐름 개요 (v1.2 Stage 체계)

v1.2 파이프라인은 **S0~S6 Stage** 체계를 따른다. 본 문서는 S0~S2(데이터 레이어)를 상세히 다룬다.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                  데이터 파이프라인 전체 흐름 (v1.2)                         │
└─────────────────────────────────────────────────────────────────────────────┘

[Stage S0: Raw Ingest — Bronze 레이어]
    │  Dataset A (Server-i CSV/JSON) + Dataset B (레이블 Excel)
    │  · Schema Registry 기반 스키마 검증 + 컬럼명 정규화
    │  · bronze_events.parquet, bronze_labels.parquet
    ▼
[Stage S1: Normalize & Parse — Silver-S1 레이어]
    │  · "1셀=다중검출" → "1행=1검출 이벤트" 분해 (3단 폴백 파서)
    │  · left_ctx / masked_pattern / right_ctx / full_context 생성
    │  · full_context_raw 비파괴 보존 + full_context_norm 파생
    │  · pk_file, pk_event 생성 (해시 기반 복합 PK)
    │  · pii_type_inferred 재추론, email_domain 추출
    │  · parse_status 기록 + Quarantine 분리
    │  · silver_detections.parquet, silver_quarantine.parquet
    ▼
[Stage S2: Feature Prep — Silver-S2 레이어]
    │  · raw_text, shape_text, path_text 생성 (멀티뷰)
    │  · tabular 피처 정규화 (log1p, is_mass_detection 등)
    │  · placeholder 비율 피처 (OOD 신호)
    ▼
[Feature Matrix → S3a RULE Labeler + S3b ML Labeler]
    │  (05_Model_Development.md 참조)
    ▼
[결과 매핑 — S5 Output]
    └── PK 기반 원본 데이터 조인 → predictions_main + prediction_evidence
```

> **Dataset C (현업 피드백 데이터):** v1.2에서 범위 제외 확정. 비정형 사유 구조화 공수 대비 효익 낮음.

## 2. 데이터 소스 분석

### 2.1 Server-i 검출 데이터의 구조

Server-i DLP 솔루션이 생성하는 검출 데이터는 다음과 같은 구성 요소를 포함한다:

**텍스트 데이터**:

| 구분 | 설명 | 특성 |
|------|------|------|
| **검출 텍스트 (원본)** | PII가 검출된 위치의 원본 텍스트 | Asterisk 처리 전 상태 |
| **검출 텍스트 (마스킹)** | Asterisk 처리된 텍스트 | 실제 PII 값 비가시화 |
| **컨텍스트** | 검출 위치 앞뒤의 텍스트 | 20-30자 내외 추정 |

**메타데이터 (Tabular)**:

| 카테고리 | 예상 필드 | 설명 |
|----------|-----------|------|
| **서버 정보** | 서버명, 서버 그룹, IP | 검출이 발생한 서버 |
| **파일 정보** | 파일 경로, 파일명, 확장자, 크기 | 검출이 발생한 파일 |
| **검출 정보** | PII 유형, 검출 패턴, 검출 위치 | 검출의 상세 내용 |
| **시간 정보** | 검출 일시, 파일 수정 일시 | 시간 관련 메타데이터 |
| **관리 정보** | 검출 ID, 파일 ID, 서버 ID | 식별자 |

### 2.2 레이블 데이터의 구조

레이블 데이터는 각 검출 건에 대한 정탐/오탐 판정 결과이다.

**레이블 스키마**:

| 필드 | 설명 | 비고 |
|------|------|------|
| **PK 컬럼 묶음 (Primary/Fallback)** | 복합 PK(`server_name+agent_ip+file_path`) 및 fallback `detection_id` | 유일성/비널성 필수 |
| **클래스 레이블** | 정탐 또는 오탐 세부 유형 | Multi-class |
| **(선택) 레이블링 일시** | 레이블링이 수행된 시점 | 데이터 품질 분석용 |
| **(선택) 레이블러 ID** | 레이블링을 수행한 담당자 | Inter-rater 분석용 |

**클래스 체계 (v1.2 Canonical)**:

```
학습 클래스 (8개)
    정탐 (TP)
        └── TP-실제개인정보

    오탐 (FP)
        ├── FP-숫자나열/코드
        ├── FP-더미데이터
        ├── FP-타임스탬프
        ├── FP-내부도메인
        ├── FP-bytes크기
        ├── FP-OS저작권
        └── FP-패턴맥락

운영 후처리 클래스 (1개 — 학습 클래스 아님)
    └── UNKNOWN  ← OOD/극도 불확실 케이스, Stage S4에서 자동 라우팅
```

> **⚠️ 패턴 종류 필드 신뢰도 이슈:** Dataset B의 `패턴 종류` 컬럼 직접 사용 금지. 이메일임에도 주민번호로 표기된 케이스 존재. → `pii_type_inferred` (검출 내역 텍스트 기반 재추론) 사용.

### 2.3 데이터 소스 간 관계

검출 데이터와 레이블 데이터는 **PK 컬럼 묶음(primary 복합 PK + fallback detection_id)**을 통해 1:1 관계로 연결된다.

**조인 시 고려사항**:

| 상황 | 원인 | 처리 방안 |
|------|------|-----------|
| **검출 O, 레이블 X** | 아직 레이블링되지 않음 | 학습 데이터에서 제외 |
| **검출 X, 레이블 O** | 데이터 불일치 | 원인 조사 필요, 레이블 무효화 |
| **중복 PK** | 데이터 오류 | 원인 조사, 중복 제거 로직 |

## 3. 데이터 수집 원칙

### 3.1 데이터 추출 전략

**전량 추출 vs 증분 추출**:

| 전략 | 설명 | 적용 상황 | 고려사항 |
|------|------|-----------|----------|
| **전량 추출** | 매번 전체 데이터 추출 | 데이터 규모 작음, 초기 구축 | 단순하지만 비효율적 |
| **증분 추출** | 변경/추가된 데이터만 추출 | 데이터 규모 큼, 운영 환경 | 상태 관리 필요, 복잡 |

**PoC 권장**: 전량 추출

PoC 단계에서는 데이터 규모가 상대적으로 작고, 파이프라인 복잡도를 낮추는 것이 우선이므로 전량 추출을 권장한다.

### 3.2 데이터 접근 방식

Server-i 데이터에 접근하는 방식은 여러 가지가 가능하다:

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **직접 DB 접근** | Server-i DB에 SQL 쿼리 | 유연한 쿼리, 실시간 접근 | DB 부하, 보안 고려 |
| **데이터 덤프** | 주기적 CSV/JSON 추출 | DB 부하 분리, 단순 | 실시간성 부족 |
| **API** | Server-i API 호출 | 표준화된 인터페이스 | API 존재 여부 확인 필요 |

**확인 필요 사항**:

- Server-i가 데이터 추출 API 또는 인터페이스를 제공하는지
- DB 직접 접근이 허용되는지, 허용 시 읽기 전용 계정 발급 가능 여부
- 데이터 덤프가 가능한지, 덤프 주기와 형식

### 3.3 Schema Registry 기반 스키마 검증 (v1.1+)

**스키마(Schema)**란 데이터의 구조적 정의로, 필드명, 데이터 타입, 제약조건 등을 포함한다.

**v1.2 Schema Registry 도입:** 스키마 불일치 시 파이프라인이 crash하는 대신, `schema_registry.yaml`을 기반으로 해당 행을 Quarantine으로 격리한다. (docs/Architecture/ §4.3 참조)

```yaml
# schema_registry.yaml 예시
dataset_a:
  required_columns: [dfile_computername, dfile_agentip, dfile_filedirectedpath,
                     dfile_inspectedcontent, dfile_inspectcontentwithcontext]
  rename_map:
    dfile_computername: server_name
    dfile_agentip: agent_ip
    dfile_filedirectedpath: file_path
    dfile_inspectedcontent: masked_hit
    dfile_inspectcontentwithcontext: full_context_raw
  on_missing_required: "quarantine"   # crash 대신 격리
  on_unknown_column: "ignore_and_log"
```

**검증해야 할 스키마 요소**:

| 요소 | 검증 내용 | 실패 시 조치 (v1.2) |
|------|-----------|---------------------|
| **필드 존재 여부** | 필수 필드가 모두 존재하는지 | Quarantine (`silver_quarantine.parquet`) |
| **데이터 타입** | 각 필드가 예상 타입인지 | 타입 변환 시도, 실패 시 Quarantine |
| **널(Null) 허용** | 필수 필드에 널이 없는지 | 레코드 Quarantine |
| **값 범위** | 수치형 필드가 유효 범위 내인지 | 이상치 처리 로직 |
| **유일성** | PK 필드가 유일한지 | 중복 처리 로직 |

### 3.4 데이터 버전 관리

**데이터 버전 관리의 필요성**:

- 모델 재학습 시 동일 데이터 사용 보장
- 모델 성능 변화의 원인 분석 (데이터 변화 vs 모델 변화)
- 규정 준수 및 감사 대응

**버전 관리 전략**:

| 전략 | 설명 | 적용 상황 |
|------|------|-----------|
| **스냅샷 저장** | 특정 시점의 전체 데이터 복사본 저장 | 데이터 규모 작음 |
| **델타 저장** | 이전 버전과의 차이만 저장 | 데이터 규모 큼 |
| **참조 저장** | 데이터는 원본 위치, 버전 메타데이터만 관리 | 데이터 변경 불가 보장 시 |

**최소한의 버전 정보**:

- 데이터 추출 일시
- 데이터 건수 (전체, 클래스별)
- 체크섬 또는 해시값 (무결성 검증용)
- 사용된 추출 쿼리 또는 필터 조건

## 4. Primary Key 기반 원본 매핑 전략

### 4.0 v1.2 PK 명명 체계

v1.2에서 PK는 두 단계로 구분된다. (docs/Architecture/ §5 참조)

| PK | 생성 방법 | 설명 |
|----|-----------|------|
| `pk_file` | hash(server_name\|agent_ip\|file_path) | 파일 단위 고유 ID |
| `pk_event` | hash(pk_file\|masked_hit\|event_time\|row_id) | 검출 이벤트 단위 고유 ID |

`pk_file`은 평가 분할(pk_file Group Split)과 파일 레벨 집계(file-level aggregation)에 사용되며, `pk_event`는 개별 이벤트 추적 및 `prediction_evidence` 조인에 사용된다.

### 4.1 설계 배경 및 필요성

**문제 상황**:

ML 모델은 원본 데이터가 아닌 변환된 Feature를 입력으로 받는다. Feature Engineering 과정에서:
- 텍스트는 TF-IDF 벡터, 키워드 존재 여부 등으로 변환
- 범주형 변수는 인코딩됨
- 여러 필드가 조합되어 파생 변수 생성

이 변환은 대부분 **비가역적(Irreversible)**이다. 즉, Feature 벡터에서 원본 텍스트나 파일 경로를 복원할 수 없다.

**비즈니스 요구사항**:

그러나 모델의 예측 결과는 비즈니스 프로세스에 연결되어야 한다:
- 정탐으로 판정된 건은 담당자에게 원본 정보와 함께 전달
- 오탐으로 판정된 건도 샘플링하여 원본 검토 필요
- 오류 분석 시 원본 데이터 확인 필수

**해결책: PK 기반 매핑**:

모든 데이터 처리 단계에서 Primary Key(복합 PK + fallback detection_id)를 유지하고, 최종 예측 결과와 원본 데이터를 PK로 조인한다.

### 4.2 PK 매핑 아키텍처

```

│                         PK 기반 매핑 아키텍처                               │

[원본 데이터 저장소]                    [ML 파이프라인]
        │                                     │
        │                                     │
   ┌────┴────┐                          ┌─────┴─────┐
   │ 원본    │                          │ Feature   │
   │ 데이터  │                          │ 데이터    │
   │         │                          │           │
   │ PK: 001 │──────────────────────────│ PK: 001   │
   │ PK: 002 │       PK 유지            │ PK: 002   │
   │ PK: 003 │──────────────────────────│ PK: 003   │
   │   ...   │                          │   ...     │
   └────┬────┘                          └─────┬─────┘
        │                                     │
        │                                     ▼
        │                              ┌─────────────┐
        │                              │   모델      │
        │                              │   추론      │
        │                              └─────┬───────┘
        │                                     │
        │                                     ▼
        │                              ┌─────────────┐
        │                              │ 예측 결과   │
        │         PK 기반 JOIN         │             │
        │◄─────────────────────────────│ PK: 001, ŷ │
        │                              │ PK: 002, ŷ │
        │                              │ PK: 003, ŷ │
        ▼                              └─────────────┘
   ┌─────────────────┐
   │  결합된 결과    │
   │                 │
   │ PK + 원본 + ŷ   │
   └────────┬────────┘
            │
            ▼
   [담당자 전달 / 후속 처리]
```

### 4.3 PK 설계 원칙

**좋은 PK의 조건**:

| 조건 | 설명 | 검증 방법 |
|------|------|-----------|
| **유일성** | 모든 레코드가 고유한 PK 값 | 중복 카운트 = 0 |
| **불변성** | 한번 할당된 PK는 변경되지 않음 | 시스템 정책 확인 |
| **비널성** | PK에 NULL 값 없음 | NULL 카운트 = 0 |
| **최소성** | 불필요하게 복잡하지 않음 | 단일 PK 우선, 필요 시 복합 PK 허용 |

**Server-i PK 전략 적합성 검토**:

운영 데이터에서 다음을 사전에 확인해야 한다:
- 복합 PK(`server_name + agent_ip + file_path`)가 실제 데이터에서 안정적으로 채워지는지
- fallback PK(`detection_id`)가 누락 없이 제공되는지
- 동일 검출 건 재처리 시 PK 값의 일관성이 유지되는지

**복합 PK 대안**:

단일 필드 유일성이 보장되지 않으면 복합 키를 primary로 사용하고, 단일 키는 fallback으로 운영한다.
본 리포지토리는 `config/feature_config.yaml > data.pk_columns`에서 primary/fallback을 함께 관리한다.

### 4.4 PK 유지 전략

**파이프라인 각 단계에서의 PK 처리**:

| 단계 | PK 처리 | 주의사항 |
|------|---------|----------|
| **데이터 로딩** | PK 컬럼 명시적 로드 | 컬럼 선택 시 누락 방지 |
| **데이터 필터링** | 필터링 후에도 PK 유지 | 인덱스 리셋 시 PK 손실 주의 |
| **Feature Engineering** | PK를 Feature로 사용하지 않음, 별도 유지 | Data Leakage 방지 |
| **데이터 분할** | Train/Val/Test 각각 PK 보존 | 분할 후에도 추적 가능 |
| **모델 학습** | PK는 모델 입력에 포함하지 않음 | 과적합 방지 |
| **모델 추론** | 추론 결과에 PK 매핑 | 순서 일치 또는 명시적 매핑 |

**흔한 실수와 방지책**:

| 실수 | 결과 | 방지책 |
|------|------|--------|
| 인덱스 리셋 시 PK 손실 | 원본 매핑 불가 | PK를 별도 컬럼으로 유지 |
| 셔플 후 순서 의존 매핑 | 잘못된 매핑 | 명시적 PK 기반 조인 |
| PK를 Feature로 사용 | Data Leakage, 과적합 | 학습 전 PK 컬럼 제거 |
| 중복 PK 미처리 | 조인 시 레코드 증가 | 조인 전 유일성 검증 |

### 4.5 원본 데이터 저장소 설계

**저장소 요구사항**:

- PK로 빠른 조회 가능 (인덱싱)
- 원본 데이터의 무결성 보장 (수정 불가 또는 버전 관리)
- 예측 결과와의 효율적 조인 지원

**저장소 옵션**:

| 옵션 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **원본 DB 직접 조회** | Server-i DB에서 직접 조회 | 별도 저장소 불필요 | DB 부하, 데이터 변경 가능성 |
| **로컬 파일 저장** | CSV/Parquet로 로컬 저장 | 단순, 빠른 접근 | 대규모 시 비효율 |
| **Key-Value Store** | Redis, SQLite 등 | 빠른 PK 조회 | 추가 인프라 |

**PoC 권장**:

로컬 파일 저장 (Parquet 형식) 권장:
- 컬럼형 저장으로 특정 컬럼만 로드 가능
- 압축 효율 좋음
- pandas와 호환성 우수
- 별도 인프라 불필요

## 5. 데이터 정제 전략

### 5.1 결측치 처리

**결측치의 유형**:

| 유형 | 설명 | 예시 |
|------|------|------|
| **MCAR** (Missing Completely at Random) | 결측이 완전히 무작위 | 데이터 전송 오류 |
| **MAR** (Missing at Random) | 결측이 다른 변수에 의존 | 특정 서버 유형에서 메타데이터 미수집 |
| **MNAR** (Missing Not at Random) | 결측이 결측값 자체에 의존 | 민감 정보일수록 마스킹 |

결측치 유형에 따라 적절한 처리 방법이 달라진다. MNAR의 경우 결측 자체가 정보를 담고 있으므로 단순 대체보다 결측 여부를 별도 Feature로 활용하는 것이 나을 수 있다.

**결측치 처리 전략**:

| 전략 | 설명 | 적용 상황 | 주의사항 |
|------|------|-----------|----------|
| **제거** | 결측 포함 레코드 삭제 | 결측 비율 낮음, MCAR | 데이터 손실 |
| **상수 대체** | 특정 값으로 대체 (예: "Unknown") | 범주형 변수 | 모델이 이를 별도 범주로 학습 |
| **통계량 대체** | 평균, 중앙값, 최빈값 | 수치형, 결측 비율 낮음 | 분산 과소추정 |
| **모델 기반 대체** | 다른 변수로 결측값 예측 | 변수 간 관계 존재 | 구현 복잡도 |
| **결측 표시 Feature** | 결측 여부를 별도 Feature로 | MNAR 의심 | Feature 수 증가 |

**본 PoC 고려사항**:

- 텍스트 결측: 컨텍스트가 없는 경우 빈 문자열로 처리할지, 레코드 제외할지 결정 필요
- 메타데이터 결측: 필드별 결측 패턴 분석 후 전략 수립
- PK 결측: 절대 허용 불가, 해당 레코드는 반드시 제외

### 5.2 이상치 탐지 및 처리

**이상치의 정의**:

이상치(Outlier)는 대부분의 데이터와 현저히 다른 값을 가진 관측치이다. 이상치는 데이터 오류일 수도 있고, 실제 존재하는 극단적 케이스일 수도 있다.

**이상치 탐지 방법**:

| 방법 | 원리 | 적용 대상 | 한계 |
|------|------|-----------|------|
| **통계적 방법** | Z-score, IQR 기반 | 정규 분포 가정 수치형 | 비정규 분포에서 부적절 |
| **도메인 기반** | 비즈니스 규칙 적용 | 명확한 유효 범위 존재 시 | 규칙 정의 필요 |
| **시각적 방법** | 히스토그램, 박스플롯 | 탐색적 분석 | 대규모 자동화 어려움 |
| **모델 기반** | Isolation Forest 등 | 고차원, 복잡한 패턴 | 해석 어려움 |

**이상치 처리 전략**:

| 전략 | 설명 | 적용 상황 |
|------|------|-----------|
| **유지** | 이상치 그대로 사용 | 실제 존재하는 극단값, Tree 기반 모델 |
| **제거** | 이상치 레코드 삭제 | 데이터 오류 확실, 비율 낮음 |
| **변환** | 로그 변환 등으로 분포 조정 | 오른쪽 꼬리 분포 |
| **캡핑** | 특정 분위수로 값 제한 | 극단값 영향 완화 |

**본 PoC 고려사항**:

Tree 기반 모델(XGBoost, LightGBM)은 이상치에 상대적으로 강건하다. 분할 기준이 순서 기반이므로 극단값이 분할 위치에 영향을 주지 않는다. 따라서 명백한 데이터 오류가 아닌 이상 이상치를 유지하는 것이 권장된다.

### 5.3 데이터 타입 표준화

**타입 표준화의 필요성**:

- 일관된 연산 보장 (문자열 "123"과 숫자 123은 다르게 처리됨)
- 메모리 효율성 (적절한 타입 사용으로 메모리 절약)
- 후속 처리 호환성 (특정 함수는 특정 타입만 지원)

**타입 변환 시 고려사항**:

| 변환 | 주의사항 |
|------|----------|
| **문자열 → 숫자** | 변환 불가 값 처리 (예: "N/A" → NaN) |
| **문자열 → 날짜** | 날짜 형식 파싱, 시간대 처리 |
| **숫자 → 범주** | 구간 정의 기준 |
| **float64 → float32** | 정밀도 손실 허용 여부 |

### 5.4 중복 데이터 처리

**중복의 유형**:

| 유형 | 설명 | 처리 |
|------|------|------|
| **완전 중복** | 모든 필드가 동일 | 하나만 유지 |
| **PK 중복** | PK만 동일, 다른 필드 상이 | 원인 조사 필수 |
| **유사 중복** | 일부 필드만 동일 | 비즈니스 규칙에 따라 판단 |

**PK 중복은 심각한 데이터 품질 문제**:

PK는 정의상 유일해야 한다. PK 중복이 발견되면:
1. 데이터 소스의 문제인지 파이프라인의 문제인지 원인 규명
2. 어떤 레코드를 유지할지 규칙 정의 (최신, 가장 완전한 것 등)
3. 근본 원인 해결

## 6. 데이터 분할 전략

### 6.1 분할의 목적과 원칙

**데이터 분할의 목적**:

| 데이터셋 | 목적 | 사용 시점 |
|----------|------|-----------|
| **Train** | 모델 학습 | 학습 과정 |
| **Validation** | 하이퍼파라미터 튜닝, 조기 종료 판단 | 학습 중 |
| **Test** | 최종 모델 성능 평가 | 모든 개발 완료 후 단 한 번 |

**핵심 원칙: 데이터 누출(Leakage) 방지**:

Test 데이터의 정보가 모델 학습에 영향을 주면 성능이 과대 추정된다. 이를 방지하기 위해:
- Test 데이터는 최종 평가 전까지 절대 열람하지 않음
- 전처리 파라미터(스케일링 평균/표준편차, 인코딩 매핑 등)는 Train에서만 계산
- Feature Selection도 Train에서만 수행

### 6.2 Stratified Sampling

**필요성**:

클래스 불균형 데이터에서 단순 무작위 분할을 하면 분할 결과의 클래스 비율이 원본과 달라질 수 있다. 특히 소수 클래스의 경우 Test 세트에 충분한 샘플이 없어 신뢰할 수 있는 평가가 어려울 수 있다.

**Stratified Sampling의 원리**:

각 클래스 내에서 독립적으로 무작위 분할을 수행하여, 분할 결과의 클래스 비율이 원본과 동일하게 유지되도록 한다.

**Multi-class Stratification**:

본 PoC는 Multi-class(8개 클래스)이므로, 모든 클래스에 대해 계층화를 적용한다. 만약 특정 클래스의 샘플이 매우 적으면 분할 자체가 어려울 수 있으므로, 최소 샘플 수 확인이 필요하다.

### 6.3 시간 기반 분할 고려

**시간 기반 분할이 필요한 경우**:

| 조건 | 설명 |
|------|------|
| 데이터에 시간 순서 존재 | 검출 일시, 파일 수정 일시 등 |
| 운영 환경이 시간 순서 예측 | 과거 데이터로 학습, 미래 예측 |
| Concept Drift 가능성 | 시간에 따라 패턴 변화 |

**시간 기반 분할 방법**:

```
[과거]                                                    [미래]
──────────────────────────────────────────────────────────────►
    │                    │                    │
    │      Train         │    Validation     │     Test
    │                    │                    │
    ├────────────────────┼────────────────────┼──────────────►
         70%                   15%                  15%
```

**주의사항**:

시간 기반 분할 시 Stratification이 자연스럽게 보장되지 않는다. 시간에 따라 클래스 분포가 변했다면 Train과 Test의 분포가 다를 수 있으며, 이는 의도적일 수도 있고 문제일 수도 있다.

**권장 접근**:

데이터에 시간 정보가 포함되어 있는지 확인하고, 포함되어 있다면:
1. 시간에 따른 클래스 분포 변화 분석
2. 시간 기반 분할과 Stratified 분할 모두 실험
3. 결과 비교 및 비즈니스 상황에 맞는 전략 선택

### 6.4 분할 비율 결정

**일반적 권장 비율**:

| 데이터 규모 | Train : Validation : Test | 근거 |
|-------------|---------------------------|------|
| 소규모 (<10K) | 60:20:20 또는 70:15:15 | Validation/Test에 충분한 샘플 확보 |
| 중규모 (10K-100K) | 70:15:15 또는 80:10:10 | 균형 잡힌 분할 |
| 대규모 (>100K) | 80:10:10 또는 90:5:5 | 작은 비율로도 충분한 샘플 |

**분할 비율 결정 시 고려사항**:

- 클래스별 최소 샘플 수: Test 세트에서 가장 작은 클래스도 통계적으로 유의미한 평가가 가능해야 함 (최소 30-50개 권장)
- 모델 복잡도: 복잡한 모델일수록 더 많은 학습 데이터 필요
- 튜닝 범위: 하이퍼파라미터 탐색 범위가 넓을수록 더 큰 Validation 세트 필요

## 7. 데이터 품질 검증 프레임워크

### 7.1 품질 검증의 계층

데이터 품질 검증은 여러 수준에서 이루어져야 한다:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        데이터 품질 검증 계층                                │
└─────────────────────────────────────────────────────────────────────────────┘

[Level 1: 기술적 검증]
    │
    ├── 파일 존재 및 접근 가능 여부
    ├── 파일 형식 유효성 (CSV/JSON 파싱 가능)
    └── 인코딩 정확성
    │
    ▼
[Level 2: 스키마 검증]
    │
    ├── 필수 컬럼 존재
    ├── 데이터 타입 일치
    └── PK 유일성
    │
    ▼
[Level 3: 값 검증]
    │
    ├── 결측치 비율
    ├── 값 범위 (수치형)
    ├── 유효 값 집합 (범주형)
    └── 형식 패턴 (날짜, 경로 등)
    │
    ▼
[Level 4: 관계 검증]
    │
    ├── 테이블 간 참조 무결성
    ├── 조인 결과 건수
    └── 비즈니스 규칙 (예: 검출일 ≤ 현재일)
    │
    ▼
[Level 5: 통계적 검증]
    │
    ├── 클래스 분포
    ├── Feature 분포 변화 (이전 배치 대비)
    └── 이상치 비율
```

### 7.2 품질 지표 정의

**핵심 품질 지표**:

| 지표 | 정의 | 기준 예시 | 위반 시 조치 |
|------|------|-----------|-------------|
| **완전성** | 결측치 비율 | 필수 필드 < 1% | 원인 조사, 레코드 제외 |
| **유일성** | PK 중복 비율 | 0% | 파이프라인 중단, 원인 규명 |
| **유효성** | 유효 값 비율 | > 99% | 무효 값 처리 규칙 적용 |
| **일관성** | 규칙 위반 비율 | < 0.1% | 규칙 위반 레코드 검토 |
| **적시성** | 데이터 최신성 | 정의된 기준 내 | 데이터 갱신 필요 |

### 7.3 검증 규칙 유형

**하드 규칙 vs 소프트 규칙**:

| 유형 | 설명 | 위반 시 조치 | 예시 |
|------|------|-------------|------|
| **하드 규칙** | 절대 위반 불가 | 파이프라인 중단 | PK NULL, 필수 필드 누락 |
| **소프트 규칙** | 위반 허용, 모니터링 | 경고, 로깅 | 결측치 비율 5% 초과 |

**규칙 정의 예시**:

| 필드 | 규칙 유형 | 규칙 내용 |
|------|-----------|-----------|
| PK 컬럼 묶음 | 하드 | NOT NULL, UNIQUE(복합키 기준) |
| 텍스트 | 소프트 | 길이 > 0 (빈 문자열 비율 모니터링) |
| 파일 경로 | 소프트 | 경로 패턴 매칭 |
| PII 유형 | 하드 | IN ('휴대폰', '이메일', '주민번호') |
| 클래스 레이블 | 하드 | NOT NULL, IN (정의된 클래스 목록) |

## 8. 파이프라인 모듈화 설계

### 8.1 모듈화의 원칙

**단일 책임 원칙 (Single Responsibility Principle)**:

각 모듈은 하나의 명확한 책임만 가진다. 예를 들어:
- 데이터 로딩 모듈: 데이터 소스에서 데이터를 읽어오는 것만 담당
- 검증 모듈: 데이터 품질 검증만 담당
- 변환 모듈: 특정 변환 로직만 담당

**인터페이스 정의**:

모듈 간 데이터 교환 형식을 명확히 정의:
- 입력 형식: 어떤 데이터를 어떤 형태로 받는지
- 출력 형식: 어떤 데이터를 어떤 형태로 반환하는지
- 오류 처리: 어떤 예외를 발생시키는지

### 8.2 파이프라인 컴포넌트

**컴포넌트 구조**:

```
[데이터 파이프라인 컴포넌트]

┌─────────────────────────────────────────────────────────────────────────────┐
│                              DataLoader                                     │
│   - 데이터 소스 연결                                                        │
│   - 데이터 추출                                                             │
│   - 기본 파싱                                                               │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                              SchemaValidator                                │
│   - 스키마 정의 로드                                                        │
│   - 스키마 검증 수행                                                        │
│   - 검증 결과 리포트                                                        │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                              DataCleaner                                    │
│   - 결측치 처리                                                             │
│   - 중복 처리                                                               │
│   - 타입 변환                                                               │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                              DataSplitter                                   │
│   - Stratified Sampling                                                     │
│   - Train/Val/Test 분할                                                     │
│   - 분할 결과 저장                                                          │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                              QualityReporter                                │
│   - 품질 지표 계산                                                          │
│   - 리포트 생성                                                             │
│   - 이력 관리                                                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 8.3 설정 기반 파이프라인

파이프라인 동작을 설정 파일로 제어하여 유연성 확보:

**설정 파일 구조 예시**:

```yaml
data_pipeline:
  source:
    type: "csv"  # 또는 "database", "api"
    path: "data/raw/dataset_a/<detection_file>.csv"
    encoding: "utf-8"
  
  schema:
    required_columns:
      - "detection_id"
      - "text_original"
      - "file_path"
      - "label"
    pk_columns:
      - "server_name"
      - "agent_ip"
      - "file_path"
      - "detection_id"  # fallback

  cleaning:
    missing_strategy:
      text_original: "empty_string"
      file_path: "drop"
    duplicate_strategy: "keep_first"
  
  splitting:
    strategy: "holdout_plus_cv"
    holdout_test_ratio: 0.2
    cv_folds: 5
    random_seed: 42
  
  quality:
    hard_rules:
      - "pk_not_null"
      - "pk_unique"
    soft_rules:
      - name: "missing_ratio"
        threshold: 0.05
```

### 8.4 오류 처리 및 로깅

**오류 처리 원칙**:

- **Fail Fast**: 복구 불가능한 오류는 즉시 중단하여 잘못된 결과 방지
- **Graceful Degradation**: 복구 가능한 오류는 처리하고 경고 로깅
- **상세 오류 정보**: 디버깅에 충분한 정보 포함

**로깅 수준**:

| 수준 | 용도 | 예시 |
|------|------|------|
| **DEBUG** | 상세 실행 정보 | 각 레코드 처리 상태 |
| **INFO** | 주요 진행 상황 | 단계 시작/완료, 건수 |
| **WARNING** | 잠재적 문제 | 소프트 규칙 위반 |
| **ERROR** | 심각한 문제 | 하드 규칙 위반, 예외 |

## 10. v1.2 자동화 확장 (S0/S1)

### 10.1 Auto-Schema-Detector

docs/Architecture/ §4.4 기준으로, 스키마 변경 시 사람이 `schema_registry.yaml`을 수동 수정하지 않고 **Auto-Schema-Detector**가 컬럼명을 자동 매핑한다.

**핵심 규칙**:

- 퍼지 매칭 임계값: 유사도 `>= 0.70`
- 자동 매핑 성공: `rename_map` 갱신 후 버전 태그 업데이트
- 매핑 실패(필수 컬럼): Quarantine 격리
- 매핑 실패(선택 컬럼): `ignore_and_log`

| 입력 상황 | Auto-Schema-Detector 처리 | 결과 |
|----------|---------------------------|------|
| 기존 컬럼과 이름 유사 (`computer_name` vs `dfile_computername`) | 퍼지 매칭으로 target 컬럼 자동 연결 | 즉시 처리 가능, 파이프라인 지속 |
| 신규 필수 컬럼인데 매핑 실패 | `unmapped_required`로 분류 | Quarantine 격리 + 알람 |
| 신규 선택 컬럼 매핑 실패 | `unmapped_optional`로 분류 | 무시 + 로그 적재 |

### 10.2 Quarantine 테이블 & 파싱 KPI

docs/Architecture/ §5.4 기준으로 `silver_detections.parquet`와 별도로 **Quarantine 테이블**(`silver_quarantine.parquet`)을 운영한다.

| 조건 | 대상 테이블 | 필수 기록 필드 |
|------|------------|----------------|
| 파싱 성공 | `silver_detections.parquet` | `parse_status="OK"` |
| 폴백 성공 | `silver_detections.parquet` | `parse_status="FALLBACK_*"` |
| 스키마 불일치/필수 필드 누락 | `silver_quarantine.parquet` | `quarantine_reason` |
| 인코딩 오류/구조 파손 | `silver_quarantine.parquet` | `quarantine_reason`, 원본 식별자 |

**운영 KPI (S1 필수):**

```text
parse_success_rate = 생성된 pk_event 수 / 원본 row 수
fallback_rate      = FALLBACK_* 이벤트 수 / 전체 이벤트 수
quarantine_count   = silver_quarantine 적재 행 수
```

**경보 기준 (v1.2):**

- `parse_success_rate < 0.95` -> 파서 구조 변화 대응 실패 신호
- `fallback_rate > 0.10` -> 포맷 변동 증가 신호
- `quarantine_count > 전월 대비 3배` -> 스키마 변화 급증 신호

### 10.3 모니터링 및 자동 조치 연계

S0/S1 지표는 운영 단계에서 **Auto-Remediation Playbook**과 연결한다.

- `quarantine_count > 3x` 발생 시 `trigger_auto_schema_detector`
- `parse_success_rate < 0.95` 발생 시 파서 폴백 확장 + 로그 자동 수집
- 조치 이력은 `monthly_metrics.json`과 함께 저장하여 감사 추적

## 부록 A: 데이터 품질 차원 정의

| 차원 | 정의 | 측정 방법 |
|------|------|-----------|
| **정확성 (Accuracy)** | 데이터가 실제 값을 올바르게 반영하는 정도 | 샘플링 검증, 교차 검증 |

| **유일성 (Uniqueness)** | 데이터가 중복 없이 유일한 정도 | 중복 비율 |
| **유효성 (Validity)** | 데이터가 정의된 형식/범위 내인 정도 | 유효 값 비율 |
| **적시성 (Timeliness)** | 데이터가 최신 상태인 정도 | 데이터 연령 |
